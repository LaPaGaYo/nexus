# Nexus v1.0.41

Date: 2026-04-20
Tag: `v1.0.41`
Status: `release_candidate`

## Summary

This patch release makes stage interaction more reliable for both hosts and
operators: support-skill recommendations now come from structured verification
signals, direct CLI runs have standardized output modes, and blocked lifecycle
stages stop advertising impossible ready-path next actions.

## What Ships

- verification matrices now persist structured `support_skill_signals` for
  design review, browser QA, benchmark, security review, Chrome attachment, and
  browser-cookie import
- completion advisors now consume those signals instead of relying on scattered
  heuristics, so `/review` and `/qa` can surface `/cso`,
  `/connect-chrome`, and `/setup-browser-cookies` through the governed contract
- blocked or refused `/frame`, `/plan`, `/handoff`, `/ship`, and `/closeout`
  now emit blocked-path advisors instead of incorrectly recommending forward
  lifecycle actions
- `bin/nexus.ts` now supports standardized `--output json-with-advisor`,
  `--output json`, and `--output human` modes, with machine-readable failure
  envelopes and runtime-owned advisor context on direct CLI execution
- generated stage skills now explicitly consume runtime-owned advisor data first
  and only fall back to the on-disk advisor artifact when the runtime payload is
  unavailable

## Notes

- This release closes the remaining gap between runtime-owned advisor contracts
  and direct CLI/operator usage. Hosts no longer need to infer support-skill
  surfacing or output semantics from prompt text alone.
- The support-skill surfacing is now explicit enough to extend safely with more
  runtime signals later, instead of growing more template-only heuristics.
