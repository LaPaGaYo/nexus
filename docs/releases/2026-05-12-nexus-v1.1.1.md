# Nexus v1.1.1

Date: 2026-05-12
Tag: `v1.1.1`
Status: `release_candidate`

## Summary

Patch hotfix for v1.1.0. Fixes a release-tarball install failure that left v1.0.50 users unable to upgrade, adds streaming output for long-running provider invocations so users see progress instead of an apparent hang, and adds a regression test that prevents the install failure from recurring.

## What Ships

### Install hotfix — release-tarball setup no longer fails on missing `.git`

- **Root cause:** v1.1.0's `package.json` `build:versions` script called `git rev-parse HEAD` inline. Release tarballs unpack into directories with no `.git`, so setup exited with `fatal: not a git repository` (exit 128).
- **Fix:** `build:versions` now invokes `scripts/build/versions.ts`, which resolves the version through three fallbacks:
  1. `git rev-parse HEAD` (dev installs)
  2. `release.json` `tag` field (release-tarball installs)
  3. literal `"release"` (no-git, no-release.json — minimal fallback)
- Installs from release tarball now complete cleanly. The runtime `.version` files contain the release tag (e.g. `v1.1.1`) instead of an empty file or build failure.

### Provider streaming — `/build`, `/review`, `/qa` no longer look like hangs

- **Root cause:** `lib/nexus/adapters/local.ts` `defaultRunCommand` accumulated all child stdout/stderr in strings and returned them only at child exit. For long-running provider invocations (claude/codex/gemini, build timeout 30 min, review/qa 15 min), users saw no output for the duration and reasonably concluded the process had hung.
- **Fix:** Added `stream_to_tty?: boolean` to `LocalCommandSpec`. When set, `defaultRunCommand` tees stdout/stderr chunks to the controller TTY in real time while still accumulating them for the returned result. `runProviderCommand` sets `stream_to_tty: true` for all three providers and prints a one-line banner before dispatch (`[nexus/local-provider] dispatching claude (timeout: up to 30 min) — streaming below:`) so the dispatch is unmistakable from a hang.
- Default remains off for internal health checks, version probes, and capability discovery so they stay quiet.

### Release-pipeline regression gate

- New test `test/nexus/release/build-versions-gitless.test.ts` exercises `scripts/build/versions.ts` in three failure modes (no `.git` + release.json with tag; no `.git` + no release.json; malformed release.json) and asserts exit 0 plus correct file contents. This runs as part of `bun test` which `nexus-release-preflight` calls, so the v1.1.0 class of bug cannot recur silently.

Plus the new gitless-install simulation:

```bash
# Smoke: simulate release-tarball setup in a no-git directory
TMP=$(mktemp -d)
cp scripts/build/versions.ts release.json "$TMP/"
mkdir -p "$TMP/runtimes/browse/dist" "$TMP/runtimes/design/dist"
cd "$TMP" && bun run versions.ts
# Expected: writes v1.1.1 to both .version files, exit 0
```

## Notes

- The same release-publish flow as v1.1.0 (`nexus-release-preflight` → `nexus-release-publish` → `nexus-release-smoke`); preflight will now run the new gitless test and catch any future reintroduction.
- v1.0.50 → v1.1.1 direct upgrade is supported (`compatibility.upgrade_from_min_version: 1.0.0`).
- Users currently stuck on v1.0.50 due to the v1.1.0 install failure can `/upgrade` to v1.1.1 directly.
- For users with accumulated `.nexus.bak-*` directories from failed v1.1.0 upgrade attempts: `rm -rf ~/.claude/skills/.nexus.bak-*` is safe (these are pre-upgrade backups, never restored automatically after success).
