# Design: Nexus Self-Learning V1 — SP1 (Signal Architecture)

> **Generated by:** SP1 brainstorming, 2026-05-12
> **Status:** ACTIVE (spec, not yet implementation)
> **Predecessor:** `docs/designs/SELF_LEARNING_V1_META.md` (v2)
> **External reviews consumed:**
> - glaocon, "Verdict on learning loop design", 2026-05-11 — 3-surface storage model
> - glaocon, "learning loop design META v1 feedback", 2026-05-11 — schema sharpening + mirroring contract
> - glaocon, "supersdes fill and entry migration", 2026-05-12 — `supersedes` semantics + lazy migration

## §1 Purpose & Scope

SP1 implements the **signal architecture** layer of the self-learning loop:

- A versioned schema (`schema_version: 2`) for learning entries with explicit
  lineage (`id`, `supersedes`, `derived_from`, `supersedes_reason`) and
  subject/writer separation (`writer_skill` vs `subject_skill` /
  `subject_stage`).
- A **three-surface storage model**: raw operational log (`learnings.jsonl`),
  per-stage provisional candidates (`learning-candidates.json`), governed
  canonical run learnings (`learnings.json`).
- An **optional mirror** from canonical back into the jsonl log under explicit
  idempotency and traceability guarantees.
- **Lazy read-time normalization** of legacy v1 entries — the existing
  `learnings.jsonl` is never rewritten.
- **Two reference implementations** wiring the design end-to-end before the
  remaining capture sites adopt the templates: `/investigate` (direct) and
  `/build` (chain).

### What SP1 does NOT ship

- LearningContextResolver / read-side ranking — that is SP6.
- Per-skill fitness scoring — that is SP2.
- Multi-provider Review Army (`/review` deeper integration) — that is SP4.
- Auto-mutation of `SKILL.md.tmpl` — that is SP3, deferred.
- A physical migrate command (`nexus learn migrate --dry-run/--write`) — that
  is a future phase after SP1 lands and resolver tests pass.

## §2 Three-surface storage model

```
Stage (build / review / qa)
   │
   │ writes provisional (chain template)
   ▼
.planning/current/<stage>/learning-candidates.json
   │
   │ /closeout reads & promotes valid candidates
   ▼
.planning/current/closeout/learnings.json   ← canonical, governed
   │
   │ optional mirror (A + D + F; see §6)
   ▼
~/.nexus/projects/<slug>/learnings.jsonl    ← raw, append-only, durable

Support skills (investigate / simplify / cso / retro)
   │
   │ direct write template
   ▼
~/.nexus/projects/<slug>/learnings.jsonl
```

Two write templates, one promotion mechanism, one optional mirror. The jsonl
log is the union of (a) direct writes from support skills and (b) optional
canonical mirrors from `/closeout`.

## §3 Schema v2

Each entry is a JSON object on its own JSONL line. All schema v2 entries
carry `"schema_version": 2`.

```json
{
  "id": "lrn_01HK8R9P5T7Z3X4N2M1Q8V6W5Y",
  "schema_version": 2,
  "ts": "2026-05-12T03:14:15.926Z",
  "writer_skill": "investigate",
  "subject_skill": "build",
  "subject_stage": "build",
  "type": "pitfall",
  "key": "n-plus-one-query-on-orders-list",
  "insight": "Orders list endpoint emits N+1 queries when serializing nested customer fields without using .includes(); test reproduces in test/api/orders.test.ts.",
  "confidence": 8,
  "evidence_type": "test-output",
  "source": "observed",
  "files": ["app/api/orders/route.ts", "test/api/orders.test.ts"],
  "cluster_id": null,
  "supersedes": [],
  "supersedes_reason": null,
  "derived_from": [],
  "last_applied_at": null,
  "mirror": null
}
```

### Field semantics

| Field | Type | Purpose | Required |
|---|---|---|---|
| `id` | `lrn_<ULID>` (30 chars total) | Stable globally-unique identifier; time-sortable | ✅ |
| `schema_version` | `2` literal | Versioning for forward migration | ✅ |
| `ts` | ISO-8601 UTC | Write timestamp; pinned in entry, never inferred | ✅ |
| `writer_skill` | string | The skill that logged this learning (renamed from `skill`) | ✅ |
| `subject_skill` | string \| `"unknown"` | The skill the learning is *about* (may differ from writer) | ✅ |
| `subject_stage` | one of `discover\|frame\|plan\|handoff\|build\|review\|qa\|ship\|closeout\|"unknown"\|null` | The lifecycle stage the learning attaches to | ✅ |
| `type` | `pattern\|pitfall\|preference\|architecture\|tool` | Category (existing) | ✅ |
| `key` | kebab-case string, 2–8 words | Dedup key when combined with `type` | ✅ |
| `insight` | string | One-sentence actionable lesson | ✅ |
| `confidence` | int 1–10 | Logger's calibrated confidence per `/learn` Iron Law 1 rubric | ✅ |
| `evidence_type` | `test-output\|code-pattern\|profile-data\|multi-run-observation\|single-run-observation\|team-consensus\|external-reference\|speculation` | Kind of evidence (not strength — see §4) | ✅ |
| `source` | `observed\|inferred\|cross-model\|user-stated\|team-consensus\|external-reference\|speculation` | How the learning was acquired | ✅ |
| `files` | string[] | Repo-relative or `~`-relative file paths | optional |
| `cluster_id` | string \| `null` | First-class clustering across multi-finding sources (used by SP4 Review Army to aggregate provider verdicts) | optional |
| `supersedes` | string[] of ids | Earlier durable entries this entry **replaces, corrects, or invalidates** | optional (default `[]`) |
| `supersedes_reason` | `correction\|stale\|contradiction\|merged\|scope-refined\|null` | Why the predecessor is no longer active | optional |
| `derived_from` | string[] of ids | Source entries (e.g., candidates) this entry was **built from** — provenance, not replacement | optional (default `[]`) |
| `last_applied_at` | ISO-8601 \| `null` | Most recent time SP6 LearningContext applied this entry as a boost (operator-state field, not write-time) | optional |
| `mirror` | object \| `null` | Set only on mirrored entries; see §6 | optional |

### Evidence strength is derived, not stored

Per glaocon V1 feedback, `evidence_type` describes **kind**, not strength.
Strength is derived by LearningContextResolver and fitness function from
`(evidence_type, confidence, source)` using a documented formula:

```
strength(entry) = clamp(1, 10,
    base[entry.evidence_type]      // test-output: 4, code-pattern: 3,
                                    // profile-data: 4, multi-run-observation: 3,
                                    // single-run-observation: 2, team-consensus: 2,
                                    // external-reference: 1, speculation: 0
  + source_modifier[entry.source]  // observed: 2, cross-model: 2,
                                    // user-stated: 1, inferred: 0
  + confidence_signal              // confidence >= 8: 2, 5-7: 1, < 5: 0
)
```

This formula lives in `lib/nexus/learning/strength.ts` and is the **single
source of truth** for strength computation across SP6 and SP2. Re-tuning the
weights re-tunes both consumers consistently.

## §4 Storage flow — two write templates

### Direct template — support skills

Used by `/investigate`, `/simplify`, `/cso`, `/retro`.

```
support skill detects learning
   │
   │ formats as schema_v2 entry
   ▼
nexus-learnings-log '<json>'
   │
   ▼
append line to ~/.nexus/projects/<slug>/learnings.jsonl
```

Support skills run outside the lifecycle and have no per-stage candidates
home. They write durable entries directly. The write path stays the existing
`nexus-learnings-log` bash helper; the helper is upgraded to inject
`id`/`schema_version`/`ts` defaults when not present.

### Chain template — canonical lifecycle stages

Used by `/build`, `/review`, `/qa`. (`/ship` deferred — too close to release
boundary, signal value low; `/closeout` is the promoter, not a candidate
writer.)

```
canonical stage handler detects learning during execution
   │
   │ formats as schema_v2 entry (no `mirror` field)
   ▼
appends to .planning/current/<stage>/learning-candidates.json (atomic write)
   │
   │ stage continues; candidate is provisional
   ▼
/closeout reads .planning/current/<stage>/learning-candidates.json for
each stage that ran in this lifecycle
   │
   │ promotion logic (§5)
   ▼
writes .planning/current/closeout/learnings.json (canonical)
   │
   │ optional mirror to jsonl (§6)
   ▼
~/.nexus/projects/<slug>/learnings.jsonl
```

`learning-candidates.json` is **NOT JSONL** — it is a JSON object:

```json
{
  "schema_version": 2,
  "stage": "build",
  "run_id": "20260512-build-abc123",
  "candidates": [
    { "id": "lrn_01H...", ... }
  ]
}
```

The wrapping object lets the file carry per-file metadata (stage, run_id)
without polluting every candidate.

## §5 `/closeout` promotion mechanism

When `/closeout` runs, it:

1. **Discovers candidate files** — walks `.planning/current/*/learning-candidates.json`
   for stages that ran in this lifecycle.
2. **Validates each candidate** — checks `schema_version: 2`, required fields
   present, `id` shape valid, `subject_stage` matches the source file's
   directory.
3. **Deduplicates within the run** — if two candidates share
   `(subject_skill, subject_stage, type, key)`, keep the latest by `ts`. The
   discarded entries are recorded as `derived_from` on the kept one.
4. **Promotes** — writes a canonical entry per promoted candidate to
   `.planning/current/closeout/learnings.json`. Canonical entries carry:
   - **new `id`** (fresh ULID — canonical and candidate are different entities)
   - **`derived_from`** populated with the candidate id(s) this was built from
   - **`writer_skill: "closeout"`** (closeout is the writer of the canonical
     entry; `subject_skill` carries the original skill)
5. **Mirrors (optional)** — per §6, if mirroring is enabled, append each
   promoted canonical entry to `learnings.jsonl` with a `mirror` block.
6. **Cleans up** — `learning-candidates.json` files remain in place during
   the active run for audit; they archive with the run when `/closeout`
   finalizes.

Canonical entries are **never** mutated. Re-running `/closeout` on the same
run produces the same canonical entries deterministically (or a clean
rewrite if the input candidates changed).

## §6 Mirror contract (Q4: A + D + F)

### A — Default opt-in

Mirror is **off by default**. Operator opts in via `~/.nexus/config.yaml`:

```yaml
learning:
  mirror_on_closeout: true   # default false
```

Rationale: keeps `learnings.jsonl` small for users who don't run the full
governed lifecycle frequently. Canonical learnings live in
`.planning/archive/runs/` regardless of mirror setting and are always read by
`nexus-learnings-search`.

### D — Nested `mirror` object

When mirrored, each entry carries:

```json
{
  "id": "lrn_01H...",
  ...
  "mirror": {
    "origin": "closeout",
    "run_id": "20260512-build-abc123",
    "source_path": ".planning/archive/runs/20260512-build-abc123/closeout/learnings.json"
  }
}
```

The nested object keeps mirror metadata grouped, makes mirrored entries
greppable (`grep '"mirror":' learnings.jsonl`), and lets unmirrored entries
omit the field entirely (`"mirror": null` or absent).

### F — Per-entry ledger for idempotency

Before mirroring, `/closeout` builds an in-memory set of
`(canonical_id, run_id)` already present in `learnings.jsonl` by streaming
the file once. New canonical entries whose `(id, run_id)` already exist are
**skipped** (no-op, no second append). New entries are appended atomically.

This protects against:
- `/closeout` re-runs producing duplicate mirrors
- Manual partial restores from backups
- Concurrent closeouts in pathological scenarios (last-write-wins is still
  safe because both writers see the same canonical content)

## §7 Lazy migration (Q6)

The existing `learnings.jsonl` is **never rewritten** by SP1. The
append-only invariant is preserved.

At read time (`nexus-learnings-search`), each line is parsed and normalized
into in-memory schema v2 before ranking/filtering:

```js
function normalizeLine(rawLine) {
  const v1 = JSON.parse(rawLine);
  if (v1.schema_version === 2) return v1;  // already v2

  // v1 (or pre-versioned) → virtual v2
  const legacyId = "legacy:" + sha256(
    [v1.ts, v1.skill, v1.type, v1.key, v1.insight, JSON.stringify(v1.files || [])].join("|")
  );
  return {
    id: legacyId,
    schema_version: 1,                              // marker: "this was migrated at read"
    ts: v1.ts,
    writer_skill: v1.skill || "unknown",
    subject_skill: "unknown",
    subject_stage: "unknown",
    type: v1.type,
    key: v1.key,
    insight: v1.insight,
    confidence: v1.confidence,
    evidence_type: "unknown",
    source: v1.source || "unknown",
    files: v1.files || [],
    cluster_id: null,
    supersedes: [],
    supersedes_reason: null,
    derived_from: [],
    last_applied_at: null,
    mirror: null,
    _migrated_at_read: true,
  };
}
```

The `legacy:<sha256>` id is deterministic so that `supersedes` and
`derived_from` references to legacy entries remain stable across re-reads.

A future `nexus learn migrate --dry-run`/`--write` command may produce a
physical `learnings.v2.jsonl` once SP1 ships and resolver tests pass.
**That command is not part of SP1.**

## §8 `supersedes` semantics (Q5)

`supersedes` means: *the older durable learning(s) should no longer be
treated as active truth.* It is **not** a generic dedup hammer.

Three explicit population paths:

1. **Write-time exact replacement** — when a new direct-template write or
   candidate has the same `(type, key, subject_skill, subject_stage)` as an
   existing durable learning AND the writer explicitly intends to **correct
   or invalidate** the previous one (not just observe the same pattern
   again). The writer sets `supersedes: [old_id]` and a
   `supersedes_reason`.
2. **Closeout canonicalization** — when `/closeout` promotes multiple
   candidates into one canonical entry, the canonical entry uses
   **`derived_from`** to reference the candidates. **`supersedes` is NOT
   used for this provenance.** (This is the distinction glaocon flagged:
   provenance ≠ replacement.)
3. **Prune-time conflict resolution** — when `/learn prune` surfaces a
   contradiction and the operator chooses "update," `/learn` appends a new
   corrected entry with `supersedes: [stale_ids]` and an appropriate
   `supersedes_reason`. The old JSONL lines are **not** rewritten.

`supersedes_reason` enum:
- `correction` — earlier entry had wrong content; corrected.
- `stale` — earlier entry was right at the time, no longer applies.
- `contradiction` — earlier entry directly contradicts; operator chose this one.
- `merged` — multiple earlier entries collapsed into one.
- `scope-refined` — earlier entry was too broad; this one narrows it.

## §9 ID algorithm

ULID (Crockford base32, 26 chars body, prefixed `lrn_`), e.g.:

```
lrn_01HK8R9P5T7Z3X4N2M1Q8V6W5Y
```

Properties relied upon:
- **Time-sortable** — first 10 chars encode ms-precision timestamp.
- **Collision-free** — last 16 chars are 80 bits of randomness.
- **Case-insensitive** — Crockford alphabet excludes ambiguous chars (I, L, O, U).

Implementation: 30-line generator in `lib/nexus/learning/id.ts` using
`crypto.randomBytes(10)`. No external dependency.

Legacy ids use the deterministic `legacy:<sha256>` scheme (§7) so that
cross-references survive multiple read passes.

## §10 Reference implementations

SP1 ships **two end-to-end reference paths** before the remaining capture
sites adopt the templates:

### Path-A — direct template (`/investigate`)

`/investigate` currently calls `nexus-learnings-log` via a bash block in its
SKILL.md. SP1 changes:

- Update the bash block to emit a schema_v2 entry (add `id`,
  `schema_version`, `writer_skill: "investigate"`, `subject_skill`,
  `subject_stage`, `evidence_type`, etc.).
- The `nexus-learnings-log` helper injects defaults for missing optional
  fields and assigns a fresh ULID if `id` is absent.
- Test: `test/learnings.test.ts` extends to cover schema_v2 round-trip via
  this skill.

### Path-B — chain template (`/build`)

`/build` does not currently capture. SP1 adds:

- A capture point in `lib/nexus/commands/build.ts` triggered by the existing
  3-strike `route to /investigate` decision (per `/build` Law 2). When the
  stop fires, `/build` writes a candidate learning to
  `.planning/current/build/learning-candidates.json` before routing.
- A `/closeout` promotion handler in `lib/nexus/commands/closeout.ts` that
  implements §5.
- The optional mirror (§6) wired into the closeout finalization step.
- Test: new `test/nexus/learning/chain-template.test.ts` covers
  candidate write → /closeout promotion → canonical write → optional
  mirror → search read.

### Remaining 5 capture sites (post-reference)

Adopt the relevant template after Path-A and Path-B prove the architecture:

- `/simplify` → direct
- `/cso` → direct
- `/retro` → direct (already writes; just schema upgrade)
- `/review` → chain
- `/qa` → chain

`/ship` is deferred — too close to release boundary, low signal value;
revisit in a follow-up phase if signal demand emerges.

## §11 Acceptance criteria

SP1 ships when **all** of the following are true:

1. **Schema** — all schema v2 fields documented in code via TypeScript types
   in `lib/nexus/learning/schema.ts`. `assertSchemaV2` validator throws on
   malformed entries.
2. **ID generator** — `generateLearningId()` produces valid ULIDs;
   `parseLearningId(id)` validates legacy and v2 forms.
3. **Strength derivation** — `computeStrength(entry)` returns 1–10 per §3
   formula. Pure function, unit-tested across all `(evidence_type, source,
   confidence)` triples.
4. **Direct template (Path-A)** — `/investigate` writes schema v2 entries.
   E2E test demonstrates write → search read → display.
5. **Chain template (Path-B)** — `/build` writes candidates, `/closeout`
   promotes to canonical with correct `derived_from`. E2E test from
   build-strike to canonical learnings.json.
6. **Optional mirror** — when `mirror_on_closeout: true`, canonical entries
   appear in `learnings.jsonl` with correct nested `mirror` block.
   Re-running `/closeout` does not duplicate (per-entry ledger guarantee).
7. **Lazy migration** — legacy v1 entries in `learnings.jsonl` are
   normalized at read by `nexus-learnings-search` and surface alongside v2
   entries. `legacy:<sha256>` ids are deterministic across re-reads.
8. **Supersedes population** — write-time exact replacement (path 1) is
   exercised by a Path-A test. Prune-time resolution (path 3) is exercised
   by `/learn prune` test. Closeout canonicalization (path 2) uses
   `derived_from` correctly per Path-B test.
9. **Backward compatibility** — search continues to return correct results
   for read-only consumers of `learnings.jsonl` that have not been updated
   for schema v2.

## §12 Test strategy

- **Unit**: schema validator, ID generation, strength derivation, legacy
  normalization — `test/nexus/learning/*.test.ts`.
- **Integration**: candidate write + closeout promotion + mirror —
  `test/nexus/learning/chain-template.test.ts`.
- **E2E**: skill template invocations through `nexus-learnings-log` and
  `nexus-learnings-search` — extension to `test/learnings.test.ts` and
  `test/skill-e2e-learnings.test.ts`.
- **Migration**: fixture file with a mix of v1 and v2 entries, search
  produces correct unified ranking — `test/nexus/learning/migration.test.ts`.
- **Idempotency**: closeout run twice on same fixtures produces identical
  jsonl content — `test/nexus/learning/mirror-idempotent.test.ts`.

## §13 Implementation questions deferred to plan stage

These are intentionally NOT decided in this spec; the implementation plan
should answer them:

- **File layout** — does `lib/nexus/learning/` get a new directory or do
  pieces land under existing `lib/nexus/observability/`? V1_META hints at
  `lib/nexus/learning-context/` for SP6; this is the chance to set the
  pattern.
- **Helper interface** — does `nexus-learnings-log` accept partial JSON and
  inject defaults, or does the caller (each skill template) construct full
  schema v2? Affects how invasive the per-skill changes are.
- **Concurrent appends** — `learnings.jsonl` may be appended by multiple
  Nexus sessions on the same project. `fs.appendFileSync` is atomic
  per-write but not across writes. Decide: file lock, line-level sequencing,
  or accept last-writer-wins.
- **Candidate write path** — does `lib/nexus/commands/build.ts` call a new
  helper `writeLearningCandidate(stage, entry)`, or do stage handlers
  directly write JSON via existing artifact-writer infrastructure?
- **TypeScript barrel** — does `lib/nexus/index.ts` (the curated barrel)
  re-export the new types? Probably yes; decide which.

## §14 Maintenance & cross-links

- **Updates V1_META §4 SP1** when SP1 ships — status: complete, link to this
  spec.
- **Triggers V1_META §9** real-world consumed row when SP1 ships:
  "SP1 landed; schema v2 in production; SP6 unblocked."
- **SP6 spec** (next sub-project) must reference §3 schema, §9 ID algorithm,
  and §6 mirror contract directly. SP6 read-side is the first consumer of
  SP1's surfaces.
- **SP2 spec** (after SP6 ships) must reference §3 strength derivation
  formula — the formula is the shared utility between SP6 ranking and SP2
  fitness scoring.

## §15 Non-goals

- LearningContextResolver implementation (SP6).
- Fitness function and skill scoring (SP2).
- Multi-provider Review Army (SP4).
- Auto-mutation of `SKILL.md.tmpl` (SP3, deferred).
- Physical migration command (`nexus learn migrate`) — future phase.
- `/ship` capture wiring (deferred; revisit if signal demand emerges).
- Schema v3 — this spec freezes v2 as the current target.
