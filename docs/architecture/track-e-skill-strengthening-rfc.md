## Goal

Raise Nexus's canonical-9 SKILL.md prose from average **2.4/5** discipline to **4.0+/5** by absorbing specific patterns from Superpowers / GSD / PM Skills into Nexus's own prose. Do this without bundling third-party skills (Model γ preserved).

When Track E lands:
- Every canonical SKILL.md has an explicit Iron Laws section
- Every canonical SKILL.md has a "Verification Before Claiming Complete" section listing specific commands
- Every canonical SKILL.md has a "Failure Mode Taxonomy" section enumerating ≥3 failure modes + recovery
- Cross-skill discipline backflows are explicit (e.g., `/simplify` finds unsafe behavior → routes back to `/review`)
- Nexus-native users (no third-party skills installed) get upstream-grade engineering discipline

---

## Strategic framing: absorb patterns, don't bundle skills

Per `skill-strength-audit.md`'s recommendation:

> "absorb specific patterns into Nexus's own prose, NOT bundle external skills"

Concretely:
- Take Superpowers `systematic-debugging`'s **Iron Law concept** → write Nexus's own Iron Law into `/build`'s SKILL.md
- Take GSD `plan-phase`'s **verification loop concept** → write Nexus's own per-task acceptance criteria into `/plan`'s SKILL.md
- Take Superpowers `verification-before-completion`'s **evidence-first concept** → write Nexus's own "Evidence Before Claims" Iron Law into every canonical
- Take Superpowers `receiving-code-review`'s **address-or-dispute pattern** → already in `/build`'s Law 3; will replicate in `/review` and `/ship`

Key constraint: **Nexus voice, not Superpowers wrapper**. Use Nexus-native concepts (runtime advisor, stage-status, advisor record, completion advisor, etc.) rather than borrowing Superpowers' framing wholesale. The /build trial run (commit `4209cc4`) demonstrates this voice: Iron Law 1 says "the runtime advisor reads what you attached, not what you remember running" — Nexus-native framing of the evidence-first principle.

This pairs with Track D-D3's Phase 5 dispatcher (`/nexus do <intent>`): users with both Nexus + third-party skills get strengthened native PLUS routing to richer external skills via Phase 5. Track E and D are complementary, not competing.

## Why this is its own track

Track A (cleanup-plan ST items), Track B (Phase 3 follow-ups), Track C (architecture audit), Track D (upstream removal + skill ecology) were all about **structural** changes — code, runtime, governance contracts. Track E is about **prose** — the words an LLM reads inside SKILL.md when deciding what discipline to apply.

Prose work has different dynamics:
- No runtime breakage risk (gen:skill-docs regenerates outputs from .tmpl)
- No test-suite blocker (skill prose isn't unit-tested by default)
- High-stakes for behavior (LLM reads prose at decision time)
- Easy to over-engineer ("voice police"; bikeshed risk)

Track E acknowledges this and runs as **independent phases** — each canonical skill is one PR, can land independently, no cross-phase blocking.

## Surface map

### Files modified per phase

```
skills/canonical/<skill-name>/SKILL.md.tmpl   ← edit (source)
skills/canonical/<skill-name>/SKILL.md         ← regenerated by gen:skill-docs
```

Plus, when generator runs, `bun run gen:skill-docs` updates host-tailored outputs (claude/codex/gemini-cli/factory paths). Per `hosts/claude/rules/skill-authoring.md`, the .tmpl is the source of truth; SKILL.md is auto-generated.

### Files NOT touched

- `lib/nexus/stage-content/<stage>/*` — runtime stage content; not skill prose
- `lib/nexus/stage-packs/*` — runtime contracts; not skill prose
- `skills/support/*` — already strong (per audit); only touch if a specific gap is identified
- `skills/safety/*` — already strong
- `skills/aliases/*` — redirect-only; no SKILL.md prose to strengthen
- Infrastructure skills (`browse`, `connect-chrome`, `setup-*`, `nexus-upgrade`) — no SDD discipline counterpart per audit

### Cross-cutting reference

The 3-pattern template from `/build` (commit `4209cc4`) is the **reference** for subsequent phases:

1. **Iron Laws block** — 2-3 hard rules; each Law = (situation → mandatory action → consequence of violation)
2. **Verification Before Claims** (within Law 1 typically) — per-change-type table of verification commands
3. **Failure Mode Taxonomy** (often as Law 2 or 3) — enumerated failure modes with recovery routing

Each phase tailors these to the skill's unique behavior surface.

### Phase template per skill

Each Phase E.N produces:

1. **Pattern selection**: which audit-identified gaps does this skill have? Which Iron Laws are appropriate?
2. **Voice draft**: write the strengthening section in Nexus voice (referencing runtime advisor, stage-status, etc.)
3. **Edit `SKILL.md.tmpl`**: insert section after `{{NEXUS_STAGE_OVERVIEW}}` and before `## Operator Checklist` (consistent placement with E.1)
4. **Regenerate**: `bun run gen:skill-docs --host codex` (and other hosts as needed)
5. **Verify**: visually check SKILL.md output; `bun run skill:check` clean
6. **Commit**: .tmpl + regenerated .md together (per skill-authoring rules)

Estimated per-phase output: 30-80 lines of new SKILL.md.tmpl content; ~30-100 lines of regenerated SKILL.md content.

## Per-phase brief (sketch)

Brief detail level for each phase below. Full brief gets written when issue is dispatched.

### Phase E.2 — `/plan` strengthening (rank 1, L)

**Audit findings**: score 1.2/5; "vague plans guarantee weak builds"; GSD `plan-phase` is strongest counterpart.

**Recommended Iron Laws**:
1. **Bite-sized task contract** — each task in `sprint-contract.md` must have (a) bounded scope (≤1 day work), (b) explicit acceptance criteria observable in artifacts, (c) verification command. Vague tasks rejected.
2. **Hypothesis before plan** — plan must articulate what could fail and how the plan mitigates. No plan ships without an explicit risk section.
3. **Plan→Build handoff is binding** — once `/plan` writes sprint-contract.md, `/build` cannot expand scope mid-execution. Scope creep returns to `/plan`.

**Verification commands**: `bun run skill:check` for plan structure; manual review of acceptance criteria specificity.

**Effort breakdown**: brief 30-45 min, .tmpl edit 30 min, regen 5 min, verification 15 min = ~1.5h.

### Phase E.3 — `/review` strengthening (rank 3, M)

**Audit findings**: score 2.0/5; "Verification-before-claim + reviewer-disagreement protocol missing".

**Recommended Iron Laws**:
1. **Two-round protocol** — Pass 1 surfaces advisories; if any block-grade, `/build` addresses; Pass 2 verifies advisories closed. Single-pass review is insufficient for ship-ready code.
2. **Specific request discipline** — when dispatching review agents, prose must specify (blocking concerns vs nice-to-have, artifacts to scrutinize, acceptance criteria upfront).
3. **Reviewer disagreement protocol** — when 2+ reviewers disagree, escalate to user via AskUserQuestion. Don't average opinions.

### Phase E.4 — `/qa` strengthening (rank 4, M)

**Audit findings**: score 2.0/5; "discipline already in `qa-only`; replicate in canonical".

**Recommended Iron Laws**:
1. **Failure mode taxonomy** (borrow `cso` format) — list 5-8 specific failure modes per stage of work + recovery routing per mode.
2. **Evidence per claim** — every QA claim ("works", "fails", "edge case found") must cite the test case + output.
3. **Stop-and-ask after 3 reproducible failures** — don't keep enumerating failures; route to `/investigate`.

### Phase E.5 — `/frame` strengthening (rank 5, M)

**Audit findings**: score 1.4/5; "PRD content contract + hypothesis framing missing".

**Recommended Iron Laws**:
1. **PRD content contract** — `prd.md` must contain ≥7 sections (problem statement, hypothesis, success criteria, non-goals, risks, alternatives considered, decision rationale). Borrow PM `prd-development` 9-section contract.
2. **Hypothesis-driven** — every PRD must state "I believe X causes Y; we will know if Z". Vague problem statements rejected.
3. **Non-goals explicit** — non-goals section must list ≥3 concrete things this work will NOT do.

### Phase E.6 — `/discover` strengthening (rank 6, M)

**Audit findings**: score 1.4/5; "Discovery anti-patterns + minimum signal bar missing".

**Recommended Iron Laws**:
1. **Discovery anti-patterns** — list 4-5 things `/discover` should NOT produce (premature solutions, scope-collapsed problems, single-stakeholder framing). Reject those upfront.
2. **Minimum signal bar** — `/discover` artifact must cite ≥2 evidence sources (user interviews, support tickets, metrics, etc.). Discovery-from-vibes rejected.
3. **Open questions explicit** — discovery brief must end with explicit open questions for `/frame` to answer.

### Phase E.7 — `/ship` strengthening (rank 7, S)

**Audit findings**: score 2.4/5; "Pre-merge gate is implicit; make it explicit".

**Recommended Iron Laws**:
1. **Pre-merge checklist explicit** — list mandatory checks (CI green, review approved, advisor stage_outcome ready, no open block-grade advisories).
2. **Decision tree** — adopt Superpowers `finishing-a-development-branch` 4-way decision (merge / keep alive / discard / split) explicitly in prose.
3. **No silent merges** — if any check fails, ship STOPS, do not auto-merge.

### Phase E.8 — `/closeout` strengthening (rank 8, S)

**Audit findings**: score 2.4/5; "Add stale-evidence detection and closeout integrity checklist".

**Recommended Iron Laws**:
1. **Stale evidence detection** — closeout reads all referenced artifacts; flags any older than 7 days OR with mismatched run_id.
2. **Closeout integrity checklist** — every milestone closeout must produce (summary, learnings, retrospective, archive index). Skipping any blocks closeout.
3. **Retroactive audit** (borrow GSD `audit-milestone`) — before archive, run audit comparing intent vs delivered.

### Phase E.9 — `/learn` strengthening (rank 9, S)

**Audit findings**: thin currently; confidence rubric missing.

**Recommended Iron Laws**:
1. **Confidence rubric** — every learning entry tagged with confidence (1-5) and evidence type. High-confidence learnings persist; low-confidence flagged for retest.
2. **Capture cadence** — learn skill triggers at end of /closeout AND on-demand. Don't lose insights to "I'll capture later".
3. **Compound-able format** — learnings stored in queryable JSONL with (topic, finding, evidence, date, confidence). Future sessions can search.

### Phase E.10 — `/handoff` strengthening (rank 10, S)

**Audit findings**: thin; "staleness check and recovery-from-blocked rule missing".

**Recommended Iron Laws**:
1. **Staleness check** — `/handoff` rejects if upstream stage status is older than 24h without explicit refresh.
2. **Recovery from blocked** — explicit protocol for re-entering `/handoff` after `/build` failed. Don't silently retry.
3. **Context handoff** (borrow GSD `pause-work`/`resume-work`) — when handoff transfers between operators, prose mandates a context dump.

## Out of scope

- **Bundling third-party skills** — Model γ preserved
- **Strengthening already-strong support skills** (investigate, simplify, cso, canary, benchmark, qa-only, design-review, careful, freeze, guard) — they're at 4.0+/5 already; touching them is fine-tuning, not Track E
- **Infrastructure skills** (browse, connect-chrome, setup-browser-cookies, setup-deploy, nexus-upgrade, unfreeze, nexus root) — no SDD discipline counterpart
- **New canonical lifecycle stages** — Track E strengthens existing 9, doesn't add new ones
- **Restructuring SKILL.md sections** — keep current section order (PREAMBLE → STAGE_OVERVIEW → Iron Laws (NEW) → Operator Checklist → ...). Don't reorder.
- **Persona injection** — audit Pattern 6 noted lack of persona in canonical; this RFC consciously leaves persona alone (potential overreach; add only if Iron Laws aren't enough)

## Risk register

| Risk | Mitigation |
|---|---|
| Strengthening makes SKILL.md too long → LLMs skim past Iron Laws | Each Iron Laws section ≤50 lines; "short and absolute on purpose" framing already in /build PoC |
| Iron Laws phrased too rigid → users override silently | Each Law has explicit dispute path (Law 3 in /build has "address OR document explicit dispute") |
| Voice creep — sounds like Superpowers wrapper not Nexus | Each phase reviews voice against `/build` PoC; Nexus voice = "runtime advisor reads what you attached" framing |
| Bikeshed risk on prose wording | Reviewer focus = "does it close the audit gap?" not "is this exact word right?" |
| Future schema changes invalidate Iron Laws | Iron Laws reference Nexus-native concepts (advisor record, stage-status) which are stable contracts |
| Canonical skills evolve, Iron Laws drift | Audit rerun after Track E lands serves as verification; future canonical changes should update Iron Laws as part of the change |
| Phase E.2 (plan) brief takes longer than estimated | L effort already accounts for brief writing; if it bloats, split into E.2a (brief) + E.2b (impl) |

## Decision points

| Decision | Resolution | Rationale |
|---|---|---|
| Phase ordering | ROI per audit (E.2 plan first; E.7-E.10 are filler) | Audit explicitly ranked |
| Need a brief per phase? | Only E.2 (L effort) needs brief; M/S phases use this RFC + audit recommendations as spec | Brief overhead not justified for small surgical edits |
| Voice constraint | Nexus-native (use advisor/stage-status framing) | Per audit recommendation; PoC validated |
| Cross-cutting templates | NOT extracted to a separate phase; each skill uses pattern in its own context | Audit's 3 patterns (Iron Law / Verification / Failure Modes) are tailored per skill, not one-size-fits-all |
| Persona injection | OUT of scope | Audit Pattern 6 noted gap, but injecting persona could overshoot; defer to future polish |
| Aliases get strengthening? | NO | Aliases redirect; no prose to strengthen |
| Already-strong skills get touched? | NO | Touching investigate / simplify / cso etc. risks regression on already-good prose |
| Infrastructure skills get touched? | NO | No SDD counterpart |
| Re-run audit after each phase? | NO; only after track complete | Per-phase audit overhead not justified; aggregate verification at end |
| Track E vs Phase 5 dispatcher coexistence | Complementary, not competing | Strengthened native + Phase 5 routing to richer external = best of both |
