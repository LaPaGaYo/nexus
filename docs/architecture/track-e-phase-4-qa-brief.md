## Goal

Raise `/qa` SKILL.md prose discipline from 2.2/5 → 4.0+/5 by adding 3 Iron Laws that codify (a) what every QA finding must carry as evidence, (b) which finding categories block `/ship` regardless of count, (c) when to stop enumerating and route to `/investigate`.

After this lands:
- A QA report cannot include findings without severity, repro, and screenshot/output evidence.
- `/qa` cannot return verdict `ready` when any P0 finding, new console error, or primary-flow health check failure exists.
- Operators stop after 3 same-root-cause failures and route to `/investigate` rather than enumerating variants.

---

## Why this is rank 4 ROI

The discipline that makes QA actually useful — severity calibration, evidence tiers, repro re-run — already lives in `qa-only` (the report-only sibling). But `/qa` (the canonical lifecycle stage) doesn't replicate it; the prose mostly delegates to runtime advisors. Per audit:

> "The rules that make QA actually useful live in `qa-only`, but the path that governs the lifecycle (`/qa`) is the thinner one."

This is voice-asymmetric: the support skill carries the discipline, the canonical skill carries the governance frame. After E.4, both carry both.

`/qa` runs after `/review` and before `/ship`. Strengthening it raises the bar at the second-to-last gate — and Law 2 makes the blocker categories visible to `/ship` directly, so `/ship` doesn't have to re-derive them.

## Iron Law design

### Law 1 — QA Finding Evidence Contract

**Discipline absorbed from:** Nexus's own `qa-only` (evidence tiers — interactive vs static) + Superpowers `verification-before-completion` (fresh evidence) + audit recommendation #1.

**Voice (Nexus-native):**

```markdown
### Law 1 — QA Finding Evidence Contract

Every finding written to `qa-report.md` MUST carry:

1. **Severity tag** — `P0` (broken primary flow / data loss / security regression) | `P1` (broken secondary flow / important UI regression) | `P2` (minor UI / non-blocking ergonomics) | `P3` (cosmetic). Untagged findings are rejected.
2. **Repro that ran in this turn** — a step-by-step that the operator executed at finding time. The repro is verified by running it once at finding time AND once at report-write time. "Should reproduce by clicking X" is not a repro; it is a guess.
3. **Evidence appropriate to category**:
   - **Interactive bugs** (broken flows, dead controls, form failures) — screenshot before action + screenshot after action + diff snapshot. All three.
   - **Static bugs** (typos, layout issues, missing images) — single annotated screenshot showing the problem.
   - **Console-only bugs** (errors / warnings without UI break) — full console line(s) including stack frame, plus the URL where it fired.
4. **No credentials** — passwords, tokens, session cookies, OAuth secrets are NEVER pasted into the report. If a finding requires authenticated repro, reference the auth method (e.g., "logged in as user@example.com") not the credential.

A finding missing severity, repro, or evidence is not a finding; it is an observation. Observations are noise; the QA report carries findings.
```

This Law replicates the `qa-only` evidence tier discipline that the audit identified as the strongest QA prose in the repo.

### Law 2 — `/ship`-Blocking Categories Are Non-Negotiable

**Discipline absorbed from:** audit recommendation #2 (explicit blocker enumeration) + Superpowers `finishing-a-development-branch` (gate before merge).

**Voice (Nexus-native):**

```markdown
### Law 2 — `/ship`-Blocking Categories Are Non-Negotiable

`/qa` MUST return verdict `not_ready` when any of these conditions hold, regardless of finding count:

1. **Any P0 finding** — broken primary user flow, data loss path, security regression. P0 in the report ≡ `not_ready` verdict. There is no "P0 but ship anyway."
2. **New console error not in baseline** — if `baseline.json` exists and the current run produces a console error not present in baseline, that is a regression. Regressions block `/ship`.
3. **Failed health check on a primary user flow** — the home / sign-in / checkout / primary CRUD page must complete its happy path without error. A failure here is `not_ready` even with no other findings.

Verdict ladder: `ready` | `ready_with_findings` (P1+/P2/P3 only, no console regressions, all primary flows green) | `not_ready` (any condition above).

`/ship` reads the verdict from the advisor record. If the verdict is `not_ready`, `/ship` cannot proceed; the operator routes back to `/build` to address blockers, then re-enters `/qa`.

The reason: shipping over a P0 trades user-visible breakage for release velocity. The trade is never worth it; reverts cost more than the revisit.
```

This Law makes the blocker contract explicit so `/ship` doesn't have to re-derive it from finding lists.

### Law 3 — Stop After 3 Same-Root-Cause Failures

**Discipline absorbed from:** RFC sketch #3 (route to `/investigate`) + `/build` Law 2 voice (3-strike stop).

**Voice (Nexus-native):**

```markdown
### Law 3 — Stop After 3 Same-Root-Cause Failures

If `/qa` finds 3+ failures that share an apparent root cause (same module, same error class, same broken interaction pattern), stop enumerating:

1. Record the 3 findings already collected with full evidence per Law 1.
2. STOP filing variants. Do not produce a 4th, 5th, 6th finding for the same root cause.
3. Route to `/investigate` with the 3 collected findings as predecessor context. `/investigate`'s 4-phase protocol (locate → analyze → hypothesize → fix) is the right tool for "what underlies this cluster."
4. `/qa` returns `not_ready` with the cluster recorded; resume after `/investigate` resolves the root.

Continuing past 3 same-root-cause findings is enumeration without insight. Each additional duplicate makes the report longer without making the fix faster. The 3rd repetition is a signal: the issue is not in the surface; it is in the substrate.
```

This Law mirrors `/build` Law 2 (3-strike stop → route to `/investigate`) — voice consistency across Track E. The trigger condition is different (failures vs fix attempts) but the stop mechanic and routing target are identical.

## Voice constraints (CRITICAL)

This is the part that distinguishes Nexus's strengthened prose from a Superpowers wrapper. Per E.1/E.2/E.3 PoC voice:

| Pattern | Don't | Do |
|---|---|---|
| Severity framing | "Use your best judgment" | "P0 in the report ≡ `not_ready` verdict. There is no 'P0 but ship anyway.'" |
| Evidence framing | "Take screenshots when useful" | "Should reproduce by clicking X is not a repro; it is a guess." |
| Stop-and-ask trigger | "After many failures, escalate" | "Continuing past 3 same-root-cause findings is enumeration without insight. The 3rd repetition is a signal: the issue is not in the surface; it is in the substrate." |
| Cross-skill flow | "Talk to the dev team" | "Route to `/investigate` with the 3 collected findings as predecessor context. `/investigate`'s 4-phase protocol (locate → analyze → hypothesize → fix) is the right tool for 'what underlies this cluster.'" |
| Mandatory rule prefix | "Iron Law:" alone | "These three rules apply to every `/qa` invocation regardless of provider, topology, or run mode. They are short and absolute on purpose — discipline that lives in qualifiers does not survive contact with the LLM at decision time." |
| Blocker framing | "Block ship if it seems serious" | "Shipping over a P0 trades user-visible breakage for release velocity. The trade is never worth it; reverts cost more than the revisit." |

Reuse the meta-justification ("short and absolute on purpose") from E.1/E.2/E.3 — voice-consistent across all Track E phases.

## Cross-Iron-Law coherence

Track E's cross-skill chain after E.4:

- **`/plan` Law 3** (binding handoff) → tasks bound to contract
- **`/build` Law 2** (3-strike stop, route to `/investigate`) → fix attempts bounded
- **`/build` Law 3** (advisories address-or-dispute) → reviewer findings can't be silently dropped
- **`/review` Law 2** (two-round on block-grade) → block-grade triggers /build re-entry, /review verifies closure
- **`/review` Law 3** (disagreement protocol) → audit-slot conflict surfaced explicitly
- **`/qa` Law 2** (`/ship`-blockers non-negotiable) → P0 / console regression / primary-flow failure ≡ `not_ready`
- **`/qa` Law 3** (3-strike stop, route to `/investigate`) → mirrors `/build` Law 2

The `/build` Law 2 and `/qa` Law 3 share both stop count (3) and routing target (`/investigate`). This is intentional: identical stop discipline across stages reduces operator cognitive load. When stuck, the answer is always the same — stop, route, resume.

## Risk register

| Risk | Mitigation |
|---|---|
| Law 1's evidence-tier requirement feels heavy for trivial findings | P3 cosmetic findings still carry single annotated screenshot — that's the lightest tier. The cost of "screenshot the typo" is seconds; the cost of vague "spelling issue somewhere" is re-investigating. |
| Law 2's "P0 ≡ not_ready" rule is rigid for genuine emergencies (hotfix bypass) | Acceptable. Hotfix bypass is `/ship` with explicit override, not `/qa` returning `ready` on a P0. The verdict honesty is what matters. |
| Law 3 fires too early (3 findings happen quickly in regression sweeps) | Law 3 scopes to "same root cause" specifically. Three independent P3 cosmetic findings in different components do NOT trigger Law 3. The clustering condition is the gate. |
| Severity ladder (P0–P3) doesn't match team conventions elsewhere | Mitigation: P0/P1 are widely understood. If team uses different scheme, follow-up phase can map. The point is severity exists and gates verdict. |
| Iron Laws section lengthens SKILL.md to ~570+ lines | Same scale as `/build` (565), `/plan` (567), `/review` (578) post-strengthening. Within design budget. |

## Decision points (resolved)

1. **Law count** — 3 (matches E.1/E.2/E.3 cadence; audit + RFC identified 3 high-leverage gaps with overlap)
2. **Cross-skill linkage** — Law 2 references `/ship`; Law 3 references `/investigate` and mirrors `/build` Law 2 explicitly. Coherence checked.
3. **Schema vs prose** — prose only. Schema work deferred.
4. **Severity vocabulary** — P0/P1/P2/P3 (industry-standard, lowest cognitive load)
5. **Verdict vocabulary** — `ready` / `ready_with_findings` / `not_ready` (mirrors `/review`'s `pass` / `pass_with_advisories` / `fail` shape; doesn't conflict because verdicts live in different stages)
6. **Evidence freshness** — folded into Law 1's repro requirement ("ran in this turn") rather than separate law; avoids duplication with `/review` Law 1.
7. **`qa-only` overlap** — `/qa` and `qa-only` now both carry evidence tier discipline. `qa-only` is more procedural (Phase 1–6 walkthrough); `/qa` is more contractual (verdict + blockers). Both useful; not redundant.
