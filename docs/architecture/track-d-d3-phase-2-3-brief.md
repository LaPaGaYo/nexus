# Track D-D3 Phase 2.3 Brief: first-party manifest catalog

**Status:** Ready for implementation. Scope: publish example `nexus.skill.yaml`
manifests and a suggestion tool for external skill authors. Nexus does not ship
or vendor the third-party skill bodies.

**Type:** Documentation catalog + generator script. Behavioral impact is
limited to author tooling; runtime SkillRegistry behavior changes only after a
user installs a skill that carries a manifest.

**Parent plan:** `docs/architecture/track-d-d3-rfc.md` Phase 3.2.c.
**Predecessor:** D3 Phase 2.a (#65) defines the schema and parser.
**Independent of:** Phase 2.2 (#74) registry consumption and Phase 2.4 (#76)
installer convenience.

**Issue:** #75.

---

## Goal

Provide a discoverable catalog of example manifests for external skill
ecosystems mentioned in the D3 audit: PM Skills, Superpowers, and GSD. The
catalog is a set of templates that external authors or marketplace maintainers
can copy, adapt, and ship alongside their own `SKILL.md` files.

This phase preserves Model gamma:

- Nexus is a router and lifecycle harness, not a warehouse.
- Nexus publishes manifest examples, not third-party skill content.
- Catalog drift cannot break Nexus runtime behavior because examples are not
  active manifests unless copied into an installed skill directory.

## Catalog layout

`docs/skill-manifests/catalog.json` is the index. It records source ecosystem,
template path, template status, and a short note about whether the template was
derived from source docs or generated by the suggestion script.

Recommended shape:

```json
{
  "schema_version": 1,
  "entries": [
    {
      "ecosystem": "pm-skills",
      "name": "example-skill",
      "template": "pm-skills/example-skill.nexus.skill.yaml",
      "status": "template",
      "notes": "Example manifest; copy next to that skill's SKILL.md before use."
    }
  ]
}
```

Template files use `.nexus.skill.yaml` suffix to avoid being mistaken for
active manifests inside this repository. Each template should be a valid
`NexusSkillManifest` after the filename is changed to `nexus.skill.yaml` and
placed next to the target skill's `SKILL.md`.

Minimum template fields:

```yaml
schema_version: 1
name: example-skill
summary: Replace with a one-line skill summary.
intent_keywords:
  - replace with a user intent phrase
classification:
  namespace: external_installed
  category: example
provenance:
  author: External skill author
  source_url: https://example.invalid
notes:
  - Template manifest. Validate against docs/skill-manifest-schema.md before shipping.
```

---

## Template authoring guidelines

1. Keep every template schema-valid.
2. Use `classification.namespace: external_installed` for PM Skills,
   Superpowers, and GSD examples.
3. Do not copy third-party `SKILL.md` content into Nexus docs.
4. Prefer broad intent phrases over brittle exact commands.
5. Use `lifecycle_stages` only when the skill purpose clearly maps to Nexus
   lifecycle stages.
6. Use `ranking.base_score` sparingly. A template can omit it and let Phase 2.2
   heuristics remain the baseline.
7. Include provenance fields when the source project and license are known from
   the source being used during implementation.

The implementation PR must document the source snapshot used to assemble the
catalog. The issue preview names the starting ecosystems and rough counts; the
PR should verify exact names from source material before committing templates.

## Suggestion script

Add:

```json
"skill:manifest:suggest": "bun run scripts/skill-manifest-suggest.ts"
```

CLI contract:

```bash
bun run skill:manifest:suggest -- --skill-md /path/to/SKILL.md
bun run skill:manifest:suggest -- --skill-md /path/to/SKILL.md --out /tmp/nexus.skill.yaml
```

The script reads a third-party `SKILL.md` and emits a stub manifest. It should
not contact the network and should not mutate the skill directory unless `--out`
is provided.

Heuristics:

- `name`: prefer frontmatter `name`, then parent directory basename.
- `summary`: prefer frontmatter `description`, then the first non-heading prose
  line, truncated to the schema limit.
- `intent_keywords`: derive from the name, summary, and first few headings;
  de-duplicate and cap to a small number.
- `lifecycle_stages`: infer only from high-confidence stage words such as
  `plan`, `review`, `qa`, `deploy`, `release`, `discover`, `requirements`.
- `classification.namespace`: always `external_installed`.
- `classification.category`: infer from source root or common words; otherwise
  omit.
- `provenance`: include only when provided through CLI flags such as
  `--source-url`, `--author`, or `--license`.

The output is a draft, not a promise. It should include a YAML comment or
`notes[]` entry that tells the author to review the manifest before shipping.

## Tests

Add `test/nexus/skill-manifest-suggest.test.ts` with these cases:

1. Frontmatter name and description produce a minimal valid manifest.
2. Missing frontmatter falls back to the parent directory name.
3. Headings and description produce de-duplicated `intent_keywords`.
4. Stage words infer `lifecycle_stages` only for supported Nexus stages.
5. Unknown or weak signals omit lifecycle stages instead of guessing.
6. `--source-url`, `--author`, and `--license` populate provenance.
7. `--out` writes the generated manifest; without `--out`, stdout is used.
8. Generated YAML round-trips through `readNexusSkillManifest()`.
9. Existing files are not overwritten unless an explicit `--force` flag is
   provided.
10. The catalog index points only to files under `docs/skill-manifests/`.

Catalog validation should be a separate helper inside the same test file or a
small script-local function. It should validate every committed template against
the Phase 2.a manifest parser.

## Acceptance criteria

1. `docs/skill-manifests/README.md` documents the catalog contract.
2. `docs/skill-manifests/catalog.json` indexes every committed template.
3. PM Skills, Superpowers, and GSD have starter template manifests based on
   implementation-time verified source names.
4. Template files are examples, not active manifests, and use a filename that
   will not be discovered by SkillRegistry inside this repo.
5. `scripts/skill-manifest-suggest.ts` emits schema-valid draft YAML from a
   `SKILL.md` input.
6. `package.json` exposes `skill:manifest:suggest`.
7. Tests cover the suggestion heuristics and catalog validation.
8. `docs/skill-manifest-schema.md` explains the catalog and suggestion flow.
9. `docs/architecture/repo-path-inventory.md` is regenerated.
10. No third-party skill body is copied into the Nexus repo.

## Out of scope

- Installing PM Skills, Superpowers, GSD, or CCB. That is Phase 2.4.
- Making the runtime registry consume catalog templates. Runtime registry only
  reads manifests next to installed `SKILL.md` files.
- Authoring manifests for Nexus-native skills. That is Phase 4.
- Implementing `/nexus do`. That is Phase 5.

---

## Risk register

| Risk | Mitigation |
|---|---|
| Catalog becomes stale | Treat it as examples only; runtime never depends on it. |
| Templates accidentally become active manifests | Use `.nexus.skill.yaml` filenames and document the copy/rename step. |
| Suggestion script over-promises correctness | Emit draft manifests and require author review. |
| External source names are guessed | Implementation PR must record the source snapshot used to build the catalog. |
| Third-party content is copied into Nexus | Store only manifests and provenance metadata, not `SKILL.md` bodies. |
