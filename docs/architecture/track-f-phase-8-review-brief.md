## Action 1 — Numbered workflow design

Insert a new `## How to run /review` section between the Iron Laws section and the Operator Checklist.

The workflow describes the typical `/review` execution loop — read upstream build evidence, dispatch the dual audit slots, collect and synthesize, route to Pass 2 if any block fires, apply the disagreement protocol if slots disagree. Steps reference Iron Laws by name rather than restating them.

7 numbered steps:

1. Read the build result (read `build-result.md`, prior `/build` `status.json`, design-contract.md if present)
2. Dispatch the dual audit slots (`codex` + `gemini` or `local_a` + `local_b`)
3. Collect audit outputs (findings + severity + provenance)
4. Synthesize per Law 1 (4 artifacts: scope summary, fresh test evidence, risk register, verdict)
5. If any finding is block-grade, route to Pass 2 protocol (AskUserQuestion gate — Action 4)
6. If audit slots disagree, apply the disagreement protocol (Law 3 escalation)
7. Write status with verdict and `pass_round`

Lifts D2 from 5/10 to ~8/10 (+3 raw, ×15 = +4.5 weighted) and D8 from 8/10 to ~9/10 (+1 raw, ×25 = +2.5 weighted). Combined: ~+7 weighted points.

## Action 3 — Typical prompts section

Insert a new `## Typical prompts` section after the Routing `Run:` block, before the Completion Advisor footer.

Three prompts:

1. **Standard dual-audit review** — "Review the change I just built on this branch." Walks Steps 1–4 + 7 → `pass` or `pass_with_advisories`, `pass_round: 1`.
2. **Block-grade finding requires Pass 2** — "Codex flagged a P1 issue — can we still ship?" Step 5 fires the AskUserQuestion gate; Pass 2 confirms closure, `pass_round: 2`.
3. **Audit slots disagree** — "Codex says block, Gemini says nit, what's the call?" Step 6 applies Law 3's enumeration; escalates via `AskUserQuestion` if speculation would otherwise be required.

Lifts D8 from ~9/10 toward 9.5/10 (already partially captured by Action 1; cumulative gain ≈ +0.5 raw, ×25 = +1.25 weighted).

## Action 4 — AskUserQuestion gate

## Voice constraints

| Pattern | Don't | Do |
|---|---|---|
| Numbered step framing | "First do X, then do Y" | "Step 1 — Read the build result. Open `.planning/current/build/build-result.md`..." |
| AskUserQuestion framing | Vague "ask user" | Explicit options reusing the governed advisor dispositions: "(a) `fix_before_qa`, (b) `continue_to_qa`, (c) `defer_to_follow_on`" |
| Cross-skill reference | "Then ship it" | Iron-Law-aware: "The advisor record's `pass_round` field tracks 1 or 2; `/ship` Law 1 reads it." |
| Workflow vs Iron Law | Either or both | Both, with explicit relationship: "Iron Laws constrain *what must be true at decision time*; this workflow defines *what to do in what order*. Both apply." |
| Trigger phrases | Aspirational | Concrete user-vocabulary: "review the change", "audit this build", "dual-review before ship" |
| AskUserQuestion duplication | Re-list dispositions in workflow AND Completion Advisor section | Reference the governed dispositions at the right moment; don't redefine them — the Completion Advisor remains source of truth |

The numbered workflow is NOT an Iron Law — it's procedure. No "MUST" or "non-negotiable" framing on the steps; the steps are guidance, the Iron Laws are constraints.

## Risk register

| Risk | Mitigation |
|---|---|
| Numbered workflow ends up duplicating Iron Law content | Workflow steps reference Iron Laws by name (e.g., "per Law 1 #1", "per Law 2", "per Law 3") rather than restating them. The two surfaces stay distinct. |
| Step 5 AskUserQuestion duplicates the existing Completion Advisor disposition section | The workflow gate references the three dispositions at the Law 2 branching moment without redefining them. The Completion Advisor section retains its full disposition enumeration verbatim — runtime advisor remains the source of truth for the actual choices. |
| 7 steps feel verbose for trivial reviews | Steps 5 and 6 are conditional (block-grade only / disagreement only). A clean Pass 1 with no disagreement uses Steps 1–4 + 7 = 5 steps end-to-end. |
| Trigger phrase "dual-review before ship" might confuse single-audit-slot operators | Step 2 explicitly notes that two slots are the floor — Law 3 cannot trigger on a single-slot review. The trigger phrase matches the dual-audit assumption that already lives in the Operator Checklist runtime content. |
| Re-generated host files (codex/factory/gemini-cli) drift from Claude canonical | `bun run skill:check` validates host parity. Run the explicit per-host regeneration commands and re-check. |
