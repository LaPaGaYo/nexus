## Iron Law design

### Law 1 βÄî Pre-Merge Readiness Gate (Five Mandatory Checks)

**Discipline absorbed from:** Superpowers `finishing-a-development-branch` (merge-readiness gate) + audit recommendation #1 + cross-skill chain (`/review` Law 1, `/qa` Law 2).

**Voice (Nexus-native):**

```markdown
### Law 1 βÄî Pre-Merge Readiness Gate (Five Mandatory Checks)

`/ship` cannot mark `release-gate-record.md` `ready` without ALL of these visible in the advisor record at decision time:

1. **Tests pass with output evidence in this turn** βÄî the verification command for the change ran in the current message AND its output is attached. Stale CI runs from a previous turn are not evidence; they describe a different state of the tree (mirrors `/build` Law 1, `/review` Law 1).
2. **CI green on the head SHA** βÄî the CI run for the exact commit being merged completes green. CI green on an earlier commit is not green for what's being shipped.
3. **`/review` verdict is current and not stale** βÄî the `pass_round` field in the advisor record is the latest review pass; subsequent commits to the branch invalidate it (per `/review` Law 2). If commits landed after Pass 2, re-enter `/review`.
4. **`/qa` verdict is `ready` or `ready_with_findings`** when `/qa` was run. `/qa` verdict `not_ready` is non-negotiable: any P0 finding, console regression, or primary-flow health check failure (per `/qa` Law 2) blocks `/ship`. There is no override.
5. **Branch is up-to-date with base** βÄî the branch contains the latest base-branch commit. Stale-base merges introduce conflicts that the gate didn't see; rebase first, then re-verify checks 1βÄ™4.

If any check fails, `/ship` returns `not_ready` and refuses to merge. The release gate is the contract for downstream `/closeout`; ambiguity here lands as broken main.
```

This Law makes the 5 readiness checks structural rather than discretionary.

### Law 2 βÄî Refuse To Ship Broken (No Silent Merges, No Force-Push, No Skip-CI)

**Discipline absorbed from:** RFC sketch #3 + audit recommendations #2 and #3.

**Voice (Nexus-native):**

```markdown
### Law 2 βÄî Refuse To Ship Broken

`/ship` MUST refuse to merge when any of these conditions hold, regardless of operator framing:

1. **No silent merges past failed checks** βÄî if any of the five readiness checks (Law 1) fails, `/ship` STOPS. The advisor record carries `release_gate: not_ready` with the failing check enumerated. The operator either fixes the failure or routes back through the relevant stage (`/build`, `/review`, `/qa`).
2. **No force-push to base or shared branches** βÄî `git push --force` to `main`, `master`, or release branches is forbidden. `/ship` operates on the ship branch only; the merge happens via PR mechanics, not by overwriting base history.
3. **No skipping CI** βÄî `[skip ci]` commit messages, `--no-verify` flags, or branch-protection bypasses are forbidden at the `/ship` layer. CI is part of the readiness gate (Law 1, check 2); skipping CI βâΓ skipping the gate.

When the user asks `/ship` to merge despite a failing check, the answer is to surface the check and explain. Bad work is worse than no work; a regression shipped past a failing check costs more than the revisit. The user can choose to re-enter the failing stage; `/ship` cannot choose to overlook it.

This rule has one exception path: an explicit, documented, time-boxed override flow that requires `AskUserQuestion` confirmation AND records the override in `release-gate-record.md` with rationale. Silent override is not an option.
```

This Law makes the merge boundary an absolute. The exception path is documented but requires explicit operator action and an artifact trail.

### Law 3 βÄî Branch Decision Tree Is Explicit

**Discipline absorbed from:** Superpowers `finishing-a-development-branch` (4-way decision) + RFC sketch #2.

**Voice (Nexus-native):**

```markdown
### Law 3 βÄî Branch Decision Tree Is Explicit

When `/ship` runs, the work on the ship branch ends in exactly one of four outcomes. The operator picks; `/ship` records the choice in `pull-request.json` and `release-gate-record.md`:

1. **Merge** βÄî readiness gate is `ready`, the PR merges to base, the branch is deleted (per repo policy). Standard happy path.
2. **Keep alive** βÄî the branch is not ready to merge but represents valid in-progress work. `/ship` records `release_gate: deferred` with a one-paragraph reason ("blocked on dependency X", "awaiting design review", "scheduled for next release window"). The branch remains; `/closeout` may not consume it yet.
3. **Discard** βÄî the work was explored, learned from, but is not shipping. `/ship` records `release_gate: discarded` with a discovery-style note (what was tried, what was learned, why this approach didn't proceed). The branch closes without merge.
4. **Split** βÄî the branch contains too much; some commits are mergeable now, others need rework. `/ship` records `release_gate: split` and routes back to `/plan` to extend the contract (per `/plan` Law 3) for the unmergeable subset, while the mergeable subset advances through Law 1.

Outcomes 2βÄ™4 are not failures; they are honest endings. The failure mode is "neither merging nor explicitly choosing one of the other three." Branches in limbo accumulate; in-flight branches without an outcome become operational debt.

`/ship` does not auto-pick an outcome. The operator picks; the artifact records the choice.
```

This Law turns "branch dies in a tab" into a recorded decision.

## Risk register

| Risk | Mitigation |
|---|---|
| Law 1 check 5 ("branch up-to-date with base") creates rebase churn for long-running branches | Acceptable. Rebase churn is operational; merge conflicts shipped to main are user-visible. The rule trades operator time for user-facing reliability. |
| Law 2's "no force-push" is too broad βÄî sometimes force-push to a topic branch is fine | Law 2 scopes to "base or shared branches" specifically. Force-push to a personal feature branch is not blocked; force-push to `main`, `master`, or release branches is. |
| Law 3's 4 outcomes feel formal for one-commit fixes | Outcome 1 (merge) is the typical case. Outcomes 2βÄ™4 are for the cases where merge isn't right; they prevent branches from dying silently. The cost of recording is seconds; the cost of operational debt accumulates. |
| `release_gate: deferred/discarded/split` may not exist in current schema | Schema work is downstream; the prose discipline ("the operator picks; the artifact records the choice") is the priority. If schema doesn't carry it yet, the operator records in `release-gate-record.md` markdown until schema catches up. |
| Iron Laws section + cross-skill references make `/ship` SKILL.md the densest in Track E | Acceptable. `/ship` is the most-consequential stage; density is appropriate. Within design budget (~580βÄ™600 lines after strengthening). |
